# AiJi · 账号体系与商业化设计文档

> 版本：1.0 · 2026-07-17
> 状态：方案设计（后端技术栈待定，见 §5 对比）
> 范围：游客/注册/付费三层用户、内置 key、权益包、云备份、与本地优先的边界

---

## 0. 设计目标与核心矛盾

**目标**：上架商店需要可统计的月活 + 商业化路径，同时守住 AiJi「本地优先」灵魂。

**三层用户**（渐进式）：
1. **游客**：不登录，自带 BYOK key，全功能可用
2. **注册免费**：登录享内置 key 限额（免自配 key 门槛），数据仍本地
3. **注册付费**：权益包——更高内置 key 额度 + 云备份同步

**核心矛盾**：本地优先 = 无服务器无账号；商业化 = 需要账号服务器。解法是**分层承诺**——数据始终本地优先，账号只管「权益凭证 + 备份通道」，不是数据中转站。

## 1. 用户分层与权益

| | 游客 | 注册免费 | 注册付费 |
|---|---|---|---|
| 登录 | 否 | 是（邮箱/手机） | 是 |
| 数据存储 | 本地 | 本地 | 本地 + 可选云备份 |
| AI 功能 | 自带 BYOK key | 内置 key（限额）+ 可自带 | 内置 key（更高限额）+ 可自带 |
| STT 功能 | 自带 BYOK key | 内置 key（限额）+ 可自带 | 内置 key（更高限额）+ 可自带 |
| 云备份同步 | ❌ | ❌ | ✅ |
| 对话检索/AI 写 | ✅（需自带 key） | ✅ | ✅ |
| 月活可统计 | ❌（匿名遥测可选） | ✅ | ✅ |
| 价格 | 免费 | 免费 | 待定（建议 ¥9-19/月 或 ¥68-128/年） |

**关键设计原则**：
- **游客不是阉割版**——自带 key 即全功能，这是「本地优先」承诺的底线
- **注册的诱惑是「省去自配 key」**——内置 key 限额是最强转化点，不付费也能用
- **付费的诱惑是「更多额度 + 云备份」**——不卖功能阉割，卖便利和扩容

## 2. 内置 key 机制（核心商业化点）

### 2.1 工作原理

用户用「内置 key」时，AI 调用走**你的代理端点**，而非直连 DeepSeek/DashScope：

```
用户 App → 你的代理(Proxy Worker) → DeepSeek/DashScope
                ↑
          扣减用户额度 + 注入你的真实 key
```

**关键**：你的真实 API key 只在代理端，**绝不下发给客户端**。客户端只持有「用户凭证（JWT）」，代理端验证凭证 + 查额度 + 注入 key 转发。

### 2.2 与现有架构的对接

现有 `SecretStorePort` + `apiKeyRef` 机制天然支持：

```ts
// Settings 扩展
interface Settings {
  // 现有 BYOK 字段...
  llmUrl?: string
  llmModel?: string
  apiKeyRef?: string       // 'llm:key' (自带) 或 'builtin:llm' (内置)
  // 新增：账号 + 权益
  account?: { userId: string; token: string; plan: 'guest'|'free'|'paid'; expiresAt?: string }
  keySource: 'builtin' | 'byok'   // 当前 AI 调用用内置还是自带
}
```

**LLM/STT 适配器分支**：
- `keySource === 'byok'`：走现有逻辑（用户 key 直连，deepSeekLlm/paraformerStt 不变）
- `keySource === 'builtin'`：走新适配器 `builtinLlm`/`builtinStt`，调你的代理端点，带用户 JWT

这复用了端口/适配器架构——新增 `BuiltinLlmPort` 适配器，不动 Domain/管线。

### 2.3 额度模型

| | 免费限额（建议） | 付费限额（建议） |
|---|---|---|
| LLM 分类 | 50 条/天 | 无限（合理上限防滥用） |
| LLM 聚合 | 10 次/天 | 无限 |
| STT 转写 | 20 分钟/天 | 200 分钟/天 |
| 云备份 | ❌ | 自动 + 手动 |

额度存服务端（防客户端篡改），客户端显示「今日剩余 N 次」。超额时引导升级或切自带 key。

### 2.4 成本控制（你扛 LLM 费用）

- 内置 key 用便宜模型（deepseek-v4-flash，单价低）
- 设日额度上限，防止滥用刷量
- 监控成本/用户，若免费用户成本 > 预期，收紧限额或引导付费
- 估算：免费用户日均 5 条 × ¥0.001/条 ≈ ¥0.005/天/用户，1000 免费用户 ≈ ¥150/月——可控

## 3. 云备份同步（付费权益）

### 3.1 设计原则：备份非同步

- **MVP 做备份，不做实时同步**——备份是「整库上传/下载」，同步是「多端实时一致」，后者复杂得多
- 备份 = 用户手动/定时把本地 Dexie+OPFS 打包上传；恢复 = 下载覆盖
- 先解决「换机不丢数据」，多端实时同步是远期

### 3.2 备份内容
- Dexie 数据库导出（entries/entryAi/categories/tags/aggregates/reminders/drafts/settings）
- OPFS 媒体 blob（音频/视频/图片）——体积大，可能需增量/分片

### 3.3 冲突处理（MVP 简化）
- 整库覆盖策略：恢复时本地库被备份覆盖（提示「将替换本地数据」）
- 不做字段级 merge——复杂且易出错
- 远期演进：按 `updatedAt` 时间戳做 last-write-wins 的条目级同步

### 3.4 与本地优先的边界
- **本地永远是 source of truth**——云备份是副本，不是主
- 用户可随时导出本地数据（现有 .zip 导出）+ 彻底删除云端备份
- 不登录用户数据完全在本地，不受任何后端影响

## 4. 月活统计（上架需求）

### 4.1 游客月活
- 游客**可统计但需同意**：首次启动匿名遥测开关（§AI 写文档前的讨论）
- 同意的游客回传 `{anonId, date, version}`，去重计月活
- 不同意的游客统计不到——这是诚实的下限

### 4.2 注册用户月活
- 登录用户每次打开 App，JWT 续期时服务端记活跃
- 精确可信，是月活的主力数据

### 4.3 对外话术
- 「N 名注册用户 + M 名匿名分享用户活跃」——诚实且有故事感
- 绝不为月活破「无服务器」承诺（统计端点 ≠ 数据服务器）

## 5. 后端技术栈对比（待定）

### 方案 A：Cloudflare Workers + D1/KV/R2

| 维度 | 评价 |
|------|------|
| 成本 | 免费额度大（10万请求/天），初期零成本 |
| 运维 | 零运维，Serverless |
| 速度 | 边缘节点全球快 |
| 开发 | Workers + D1(SQLite) + KV(额度) + R2(备份存储)，JS/TS 全栈 |
| 国内访问 | ⚠️ Cloudflare 国内访问不稳定，可能影响国内用户体验 |
| 合规 | 数据境外，国内上架商店可能要求说明 |

### 方案 B：国内云函数（腾讯 SCF / 阿里 FC）+ COS + 数据库

| 维度 | 评价 |
|------|------|
| 成本 | 有免费额度，超了按量付费，比 CF 贵 |
| 运维 | 略重（函数 + API 网关 + 数据库 + 对象存储） |
| 速度 | 国内访问稳定快 |
| 开发 | Python/Node 云函数 + MySQL/PostgreSQL + COS |
| 国内合规 | ✅ 数据境内，国内商店上架顺利，备案可行 |
| 备份存储 | COS（对象存储，存 OPFS 媒体合适） |

### 方案 C：混合（推荐）

- **国内用户**：国内云函数（合规 + 速度）
- **海外用户**：Cloudflare（成本低）
- 账号体系统一，按地域路由到不同后端
- 适合后期；MVP 先单选

### 推荐
- **大赛/上架初期（国内为主）**：方案 B 国内云函数——合规优先，国内商店上架无障碍
- **若预算极紧且先不考虑国内合规**：方案 A Cloudflare 起步，后期再迁
- **建议**：MVP 走 B，因为你要上华为 AppGallery + 安卓商店，国内合规是硬要求

## 6. 数据模型扩展

### 6.1 客户端（Dexie）

```ts
// Settings 扩展（见 §2.2）
// 新增 accounts 不需要本地表——账号信息存 Settings.account

// 新增：备份元数据
interface BackupMeta {
  id: string
  uploadedAt: string
  sizeBytes: number
  entryCount: number
  version: number
}
```

### 6.2 服务端

```
users:        id, email, phone, passwordHash, plan, createdAt
quotas:       userId, date, llmUsed, sttUsed, aggUsed  (日额度，按日重置)
tokens:       userId, refreshToken, expiresAt           (JWT 刷新)
backups:      userId, backupId, uploadedAt, size, r2Key (备份元数据，blob 存 R2/COS)
audit_log:    userId, action, ts                        (付费/额度变更审计)
```

### 6.3 代理端点（内置 key 通道）

```
POST /api/llm/chat      → 验 JWT + 扣额度 + 注入真实 key 转发 DeepSeek
POST /api/stt/transcribe → 同上，转发 DashScope
POST /api/auth/register  → 注册
POST /api/auth/login     → 登录，返 JWT
POST /api/auth/refresh   → 续期
POST /api/quota          → 查额度
POST /api/backup/upload  → 上传备份（付费）
POST /api/backup/download→ 下载备份（付费）
POST /api/telemetry      → 匿名活跃回传（游客可选）
```

## 7. 上架合规要点

### 7.1 国内商店必需
- [ ] **ICP 备案**（有后端域名就要，~2 周）
- [ ] **软件著作权**（安卓商店必需，~1 月）
- [ ] **隐私政策**（强化版：说明账号、内置 key 代理、云备份各传什么）
- [ ] **用户协议**（含付费条款、退款政策）
- [ ] **增值电信业务许可**？若付费可能涉及 ICP 经营性备案——需确认（个人开发者可能需挂靠企业）
- [ ] **内容安全**：内置 key 通道转发用户内容到 LLM，需有内容过滤/审计（国内合规要求）

### 7.2 隐私政策更新（相比纯本地版增加）
1. 账号：注册需邮箱/手机号，用于登录和权益
2. 内置 key 代理：用内置 key 时，条目内容经你的代理转发到 LLM 提供商，**代理不存储内容**（只记额度）
3. 云备份：付费用户上传的备份数据加密存储，你可应要求删除
4. 自带 key：不经手，同现有

### 7.3 iOS 审核
- 付费须走苹果内购（30% 抽成）——要么 iOS 涨价覆盖，要么 iOS 不支持付费（仅注册免费）
- BYOK 审核说明同前

## 8. 与本地优先的边界（设计红线）

| | 游客 | 注册免费 | 注册付费 |
|---|---|---|---|
| 不登录能用全部功能？ | ✅（自带 key） | — | — |
| 数据是否本地优先？ | ✅ | ✅ | ✅（云是备份非主） |
| 后端是否碰用户内容？ | 否 | 仅代理转发（不存） | 代理转发 + 备份存储（加密） |
| 能否随时导出/删除？ | ✅ | ✅ | ✅（含云端） |

**红线**：
1. 游客永远全功能（自带 key）——不强制注册
2. 本地永远是 source of truth——云只是备份
3. 后端不存储用户条目内容（代理只转发 + 记额度，备份是用户主动上传的加密包）
4. 用户可随时删除账号 + 云端全部数据

## 10. 风险与决策点

| 风险/决策 | 说明 |
|-----------|------|
| 内置 key 成本失控 | 设日额度上限 + 监控成本/用户 + 用便宜模型 |
| 付费用户少、免费成本高 | 免费额度要克制，引导自带 key 或付费 |
| 云备份冲突 | MVP 整库覆盖，不做字段级 merge |
| 国内合规（经营性 ICP） | 付费可能需企业资质，个人开发者需确认或挂靠 |
| 后端技术栈选择 | 推荐 B 国内云函数（合规优先），见 §5 |
| iOS 付费抽成 | iOS 走内购或仅免费，见 §7.3 |
| 与本地优先的口碑 | 守住 §8 红线，宣传突出「数据始终本地，云只是可选备份」 |
