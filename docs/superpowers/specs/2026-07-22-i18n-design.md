# AiJi 多语言（i18n）设计 — 中文 / English

> 日期：2026-07-22 · 状态：已批准（方案 A + 范围「UI + 新 AI 输出」由用户拍板，细节授权 lead 自主）
> 范围铁律（用户拍板）：**UI 全部双语 + LLM 新生成输出跟随界面语言；历史已生成数据保持原语言不重译。**

## 1. 目标与非目标

**目标**
- App 全部 UI 文案中/英双语，运行时切换即时生效（不刷新）。
- 首次打开（onboarding）顶部可选语言，默认按系统语言预选（`zh*` → 中文，其余 → English）。
- 设置页「语言 / Language」随时可改。
- LLM 提示词（classify / 问答 / 记忆提取 / 聚合摘要）按当前语言生成，英文用户得到的涌现分类名、摘要、回答都是英文。
- 服务器错误（AUTH_*/配额/geocode）在客户端按错误码映射成本地化文案，英文用户看不到中文报错。

**非目标（本期不做）**
- 已生成的历史数据（旧分类名/摘要）重新翻译。
- seed 示例数据翻译（原型样例，用户主动导入）。
- Android 原生层（应用名、权限弹窗）本地化。
- 第三语言（字典结构已留扩展位，加语言 = 加一份字典文件 + Lang 联合加成员）。

## 2. 核心架构（方案 A：轻量自研）

### 2.1 语言单例 `src/app/currentLang.ts`（零依赖，镜像 currentOwner.ts 模式）

```ts
export type Lang = 'zh' | 'en'
export function detectLang(): Lang        // navigator.language.startsWith('zh') ? 'zh' : 'en'
export function getCurrentLang(): Lang    // 模块 init = detectLang()，hydrate 后覆写
export function setCurrentLang(l: Lang): void
```

**为何必须零依赖单例**：`t()` 要在非 React 上下文用（helpers、适配器、LLM 提示词构建），若直接读 uiStore 则形成 ESM 环（store → di → adapters → i18n → store）。与 currentOwner 同构，已验证可行。

### 2.2 字典 `src/app/i18n/`

扁平点路径 key，按屏分片文件（防撞 + 并行）：

```
src/app/i18n/
  zh/ home.ts  capture.ts  detail.ts  categories.ts  summary.ts  search.ts
      settings.ts  onboarding.ts  chat.ts  login.ts  drafts.ts  trash.ts
      reminders.ts  feedback.ts  common.ts  index.ts   ← index 合并所有片段 as const
  en/ （同构同名文件）                    index.ts   ← 类型约束 = zh 的全 key
  index.ts     ← t() / useT() / I18nKey 类型
```

```ts
// zh/home.ts
export const home = {
  'home.empty.title': '还没有记下任何东西',
  'home.todayCount': '今天 {count} 条',
} as const

// zh/index.ts
import { home } from './home'
// ... 全部片段
export const zh = { ...common, ...home, ...capture, /* … */ } as const
export type I18nKey = keyof typeof zh

// en/index.ts —— 类型强制与 zh key 集合一致：缺 key / 多 key 都 typecheck 挂
export const en: { [K in I18nKey]: string } = { ...commonEn, ...homeEn, /* … */ }
```

en 完整性由 `tsc` 兜底，不靠运行时测试。

### 2.3 `t()` 与 `useT()`

```ts
// src/app/i18n/index.ts
export function t(key: I18nKey, params?: Record<string, string | number>): string
//   读 getCurrentLang() → 字典取词 → {name} 插值。非 hook，任何上下文可调。
export function useT(): typeof t
//   useUiStore(s => s.settings.language) 订阅触发重渲，返回 t。
```

React 组件一律 `const t = useT()`；非组件（helpers/适配器/prompt 构建）直接 `import { t }` 或读 `getCurrentLang()` 选提示词版本。

### 2.4 持久化与首启检测

- `domain/types.ts` Settings 加 `language?: Lang`。
- store `setSettings` 时若 language 变化 → `setCurrentLang()` 同步单例。
- boot hydrate 后：`settings.language ?? detectLang()` → 写回 settings（一次性固化，之后系统语言变了不跟着变，用户可手动改）。
- onboarding 顶部语言分段控件（中文 / English），点击即 `setSettings({language})`，整屏文案即时切换（onboarding 全部走 `useT()`）。

## 3. LLM 提示词双语

`openAiCompatLlm.ts` 的 4 个提示词构建点各出双语版本，按 `getCurrentLang()` 选择：

| 构建点 | 改动 |
|---|---|
| `buildPrompt`（classify，含 VLM fallback） | 系统提示双语 + 末尾约束「用简体中文输出所有分类名/摘要」/ "Output all category names and summaries in English" |
| `buildAnswerPrompt`（问答） | 双语 + 回答语言约束（用户记忆原文注入，不翻译） |
| `buildExtractMemoryPrompt`（记忆提取） | 双语 + 记忆内容语言跟随当前语言 |
| aggregate/摘要提示词 | 双语 + 输出语言约束 |

builtinLlm 走同一批 builder（已导出），自动继承。用户消息原文不翻译直接注入。

## 4. 服务器错误客户端本地化

`httpAuth.parseAuthError` 已把后端 `{error:'AUTH_409', message:'<中文>'}` 结构化为 `Error('AUTH_409:…')`。新增 `src/app/i18n` 错误码映射表（`errors.zh/en` 片段），UI 展示层按 `err.message.split(':')[0]` 查映射，查不到回落原文。覆盖：AUTH_400/401/409/429、配额 QUOTA_429、GEOCODE_*。登录/注册/绑定/设置配额页接入。

## 5. 日期与数字

- `Intl.DateTimeFormat(lang === 'zh' ? 'zh-CN' : 'en-US', …)` 替换手写「X月X日」格式。
- 「今天/昨天/周X」相对标签入字典（`date.today/yesterday` + weekday 用 Intl）。
- home helpers 的 `groupLabel/topDateLabel/dateKey` 签名加 lang 或内部读 getCurrentLang()。

## 6. 并行改造方案（防撞）

**Lead 先行（顺序）**：
1. `currentLang.ts` + `src/app/i18n/` 骨架（含全部 15 屏 + common + errors 的空片段文件，barrel 就位，t/useT 可用）
2. `Settings.language` + setSettings 同步 + boot 检测固化
3. settings 屏「语言」行 + LanguageSheet；onboarding 语言分段控件
4. LLM 提示词双语化 + 错误码映射

**Fan-out（每子智能体一个屏目录，不碰共享文件）**：
- 每个 agent 只写：自己的 `src/ui/screens/<x>/**` + 自己的两个片段文件 `src/app/i18n/{zh,en}/<x>.ts`（lead 已建空壳，key 集合 agent 自定， barrel 已 import）。
- components/layout（NavBottom/Statusbar/Sheet/EmptyState 等）归 common 片段，派一个 agent。
- 自检：`npx tsc -p tsconfig.app.json`（禁 `npm run typecheck`，并发竞态铁律）。

**收口（lead）**：全量 typecheck + 单测 + reviewer agent（静态审 + 浏览器 390×844 双语 e2e）→ commit → bump rc → push。

## 7. 测试

- **单测**：`t()` 插值/缺参；`detectLang` 分支；en 完整性（typecheck 即测试）；错误码映射命中/回落；setSettings 切语言 → currentLang 同步。
- **e2e（reviewer，390×844）**：① 全新 profile → onboarding 语言控件预选正确、切换即时生效；② 设置切 English → home/导航/空态全英文；③ 英文态记一条 → classify 分类名英文；④ 英文态登录输错密码 → 英文报错；⑤ 切回中文全恢复。

## 8. 风险

| 风险 | 对策 |
|---|---|
| 568 处字符串漏翻 | typecheck 只保 key 存在，不保「该用 t 的地方用了」→ reviewer 逐屏中英双跑肉眼验收 + e2e 截图 |
| 片段文件 key 冲突（两屏起了同名 key） | key 以屏名前缀（`home.`/`chat.`），common 用 `common.`；typecheck 对 as const 合并重复 key 会报 |
| 切换语言后 LLM 并发请求串语言 | 提示词构建时读 getCurrentLang() 快照，单次请求内一致即可，可接受 |
| onboarding 未 hydrate 完成时语言闪烁 | currentLang 模块 init 即 detectLang() 同步有值，hydrate 后仅当用户曾手动选择才覆写 |
